39  /**
40   * <h1>LogoutRequestParser</h1>
41   * <p>
42   * This class parses a Logout Request message defined in the iSCSI Standard (RFC3720).
43   * <p>
44   * The Logout Request is used to perform a controlled closing of a connection.
45   * <p>
46   * An initiator MAY use a Logout Request to remove a connection from a session or to close an entire session.
47   * <p>
48   * After sending the Logout Request PDU, an initiator MUST NOT send any new iSCSI requests on the closing
49   * connection. If the Logout Request is intended to close the session, new iSCSI requests MUST NOT be sent on
50   * any of the connections participating in the session.
51   * <p>
52   * When receiving a Logout Request with the reason code of "close the connection" or "close the session", the
53   * target MUST terminate all pending commands, whether acknowledged via ExpCmdSN or not, on that connection or
54   * session respectively.
55   * <p>
56   * When receiving a Logout Request with the reason code "remove connection for recovery", the target MUST
57   * discard all requests not yet acknowledged via ExpCmdSN that were issued on the specified connection, and
58   * suspend all data/status/R2T transfers on behalf of pending commands on the specified connection.
59   * <p>
60   * The target then issues the Logout Response and half-closes the TCP connection (sends FIN). After receiving
61   * the Logout Response and attempting to receive the FIN (if still possible), the initiator MUST completely
62   * close the logging-out connection. For the terminated commands, no additional responses should be expected.
63   * <p>
64   * A Logout for a CID may be performed on a different transport connection when the TCP connection for the CID
65   * has already been terminated. In such a case, only a logical "closing" of the iSCSI connection for the CID
66   * is implied with a Logout.
67   * <p>
68   * All commands that were not terminated or not completed (with status) and acknowledged when the connection
69   * is closed completely can be reassigned to a new connection if the target supports connection recovery.
70   * <p>
71   * If an initiator intends to start recovery for a failing connection, it MUST use the Logout Request to
72   * "clean-up" the target end of a failing connection and enable recovery to start, or the Login Request with a
73   * non-zero TSIH and the same CID on a new connection for the same effect (see Section 10.14.3 CID). In
74   * sessions with a single connection, the connection can be closed and then a new connection reopened. A
75   * connection reinstatement login can be used for recovery (see Section 5.3.4 Connection Reinstatement).
76   * <p>
77   * A successful completion of a Logout Request with the reason code of "close the connection" or "remove the
78   * connection for recovery" results at the target in the discarding of unacknowledged commands received on the
79   * connection being logged out. These are commands that have arrived on the connection being logged out, but
80   * have not been delivered to SCSI because one or more commands with a smaller CmdSN has not been received by
81   * iSCSI. See Section 3.2.2.1 Command Numbering and Acknowledging. The resulting holes the in command sequence
82   * numbers will have to be handled by appropriate recovery (see Chapter 6) unless the session is also closed.
83   * <p>
84   * The entire logout discussion in this section is also applicable for an implicit Logout realized via a
85   * connection reinstatement or session reinstatement. When a Login Request performs an implicit Logout, the
86   * implicit Logout is performed as if having the reason codes specified below:
87   * <p>
88   * <table border="1">
89   * <tr>
90   * <th>Reason code</th>
91   * <th>Type of implicit Logout</th>
92   * </tr>
93   * <tr>
94   * <td>0</td>
95   * <td>session reinstatement</td>
96   * </tr>
97   * <tr>
98   * <td>1</td>
99   * <td>connection reinstatement when the operational ErrorRecoveryLevel < 2</td>
100  * </tr>
101  * <tr>
102  * <td>2</td>
103  * <td>connection reinstatement when the operational ErrorRecoveryLevel = 2</td>
104  * </tr>
105  * </table>
106  * <p>
107  * <h4>TotalAHSLength and DataSegmentLength</h4> For this PDU TotalAHSLength and DataSegmentLength MUST be
108  * <code>0</code>.
109  * <p>
110  * <h4>ExpStatSN</h4> This is the last ExpStatSN value for the connection to be closed.
111  * <p>
112  * <h4>Implicit termination of tasks</h4> A target implicitly terminates the active tasks due to the iSCSI
113  * protocol in the following cases:
114  * <p>
115  * <ol type="a">
116  * <li>When a connection is implicitly or explicitly logged out with the reason code of "Close the connection"
117  * and there are active tasks allegiant to that connection.</li>
118  * <li>When a connection fails and eventually the connection state times out (state transition M1 in Section
119  * 7.2.2 State Transition Descriptions for Initiators and Targets) and there are active tasks allegiant to
120  * that connection.</li>
121  * <li>When a successful recovery Logout is performed while there are active tasks allegiant to that
122  * connection, and those tasks eventually time out after the Time2Wait and Time2Retain periods without
123  * allegiance reassignment.</li>
124  * <li>When a connection is implicitly or explicitly logged out with the reason code of "Close the session"
125  * and there are active tasks in that session.</li>
126  * </ol>
127  * If the tasks terminated in any of the above cases are SCSI tasks, they must be internally terminated as if
128  * with CHECK CONDITION status. This status is only meaningful for appropriately handling the internal SCSI
129  * state and SCSI side effects with respect to ordering because this status is never communicated back as a
130  * terminating status to the initiator. However additional actions may have to be taken at SCSI level
131  * depending on the SCSI context as defined by the SCSI standards (e.g., queued commands and ACA, in cases a),
132  * b), and c), after the tasks are terminated, the target MUST report a Unit Attention condition on the next
133  * command processed on any connection for each affected I_T_L nexus with the status of CHECK CONDITION, and
134  * the ASC/ASCQ value of 47h/7Fh - "SOME COMMANDS CLEARED BY ISCSI PROTOCOL EVENT" - etc. - see [SAM2] and
135  * [SPC3]).
136  * <p>
137  * 
138  * @author Volker Wildi
139  */

145     /**
146      * This enumeration defines all the Logout Reasons Codes, which are allowed
147      * in a iSCSI Logout Request message (RFC3720). All other values are
148      * reserved.
149      * <p>
150      * All other values are reserved.
151      * <p>
152      */
153     public static enum LogoutReasonCode {
154 
155         /**
156          * Close the session. All commands associated with the session (if any)
157          * are terminated.
158          */
159         CLOSE_SESSION((byte)0),
160 
161         /**
162          * Close the connection. All commands associated with connection (if
163          * any) are terminated.
164          */
165         CLOSE_CONNECTION((byte)1),
166 
167         /**
168          * Remove the connection for recovery. Connection is closed and all
169          * commands associated with it, if any, are to be prepared for a new
170          * allegiance.
171          */
172         CONNECTION_RECOVERY((byte)2);
173 
174         private final byte value;
175 
176         private static Map<Byte, LogoutReasonCode> mapping;
177 
178         static {
179             LogoutReasonCode.mapping = new HashMap<Byte, LogoutReasonCode>();
180             for (LogoutReasonCode s : values()) {
181                 LogoutReasonCode.mapping.put(s.value, s);
182             }
183         }
184 
185         private LogoutReasonCode(final byte newValue) {
186 
187             value = newValue;
188         }

194          */

325     /** {@inheritDoc} */
326     @Override
327     protected final void deserializeBytes1to3(final int line) throws InternetSCSIException {
328 
329         reasonCode =
330             LogoutReasonCode
331                 .valueOf((byte)((line & Constants.SECOND_BYTE_MASK) >>> Constants.TWO_BYTES_SHIFT));
332         Utils.isReserved(line & Constants.LAST_TWO_BYTES_MASK);
333     }
334 
335     /** {@inheritDoc} */
336     @Override
337     protected final void deserializeBytes20to23(final int line) throws InternetSCSIException {
338 
339         connectionID = (short)((line & Constants.FIRST_TWO_BYTES_MASK) >>> Constants.TWO_BYTES_SHIFT);
340         Utils.isReserved(line & Constants.LAST_TWO_BYTES_MASK);
341     }

346     /** {@inheritDoc} */
347     @Override
348     protected final void checkIntegrity() throws InternetSCSIException {
349 
350         String exceptionMessage;
351         do {
352             Utils.isReserved(logicalUnitNumber);
353 
354             if (reasonCode == LogoutReasonCode.CLOSE_SESSION && connectionID != 0) {
355                 exceptionMessage = "The CID field must be zero, if close session is requested.";
356                 break;
357             }
358 
359             if (protocolDataUnit.getBasicHeaderSegment().getTotalAHSLength() != 0) {
360                 exceptionMessage = "TotalAHSLength must be 0!";
361                 break;
362             }
363 
364             if (protocolDataUnit.getBasicHeaderSegment().getDataSegmentLength() != 0) {
365                 exceptionMessage = "DataSegmentLength must be 0!";
366                 break;
367             }
368 
369             // message is checked correctly
370             return;
371         } while (false);
372 
373         throw new InternetSCSIException(exceptionMessage);
374 
375     }
